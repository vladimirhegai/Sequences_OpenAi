# HACKATHON_RULES.md

## OpenAI Build Week 2026

This repository is being developed for OpenAI Build Week.

All coding agents working on this project should optimize for a polished, reliable hackathon submission built meaningfully with Codex and GPT-5.6.

The Official Rules and Devpost page are the source of truth.

---

## Project

The project is **Sequences**, an AI-native motion-design editor.

The Build Week experience should let a user:

1. Provide a prompt and product image or screenshot.
2. Generate a professional motion-design composition.
3. Receive a real editable timeline with layers, scenes, properties, and keyframes.
4. Inspect the changes generated by AI.
5. Request revisions without regenerating the entire project.
6. Render and preview the final result.

Target audience:

- SaaS teams
- product marketers
- product designers
- motion designers
- founders creating launch videos

Primary track:

```text
Work and Productivity
````

Primary value proposition:

> Sequences turns product visuals and creative intent into polished motion design with a real, inspectable, editable timeline.

---

## Required technologies

The project must meaningfully use:

* Codex to build the project
* GPT-5.6 inside the working product

GPT-5.6 cannot be decorative or limited to basic copy generation.

Appropriate GPT-5.6 responsibilities include:

* analyzing an uploaded screenshot
* understanding visual hierarchy
* deciding what elements should be emphasized
* planning scenes and transitions
* generating motion direction
* producing structured timeline commands
* revising an existing composition
* critiquing pacing, hierarchy, spacing, and readability
* proposing targeted fixes

Preferred system:

```text
Prompt + image
    ↓
GPT-5.6 visual and creative reasoning
    ↓
Structured scene plan
    ↓
Validated timeline commands
    ↓
Sequences editor
    ↓
Editable composition
    ↓
GPT-5.6 critique and revision
```

Do not fake live generation with a hardcoded or prerecorded result.

## Judging criteria

The four criteria are equally weighted.

### Technological Implementation

The implementation should feel non-trivial and technically credible.

Prioritize:

* structured model outputs
* typed timeline commands
* schema validation
* inspectable AI diffs
* safe and deterministic mutations
* targeted revisions
* undo and redo support
* error handling
* reliable rendering
* tests around core command behavior

### Design

The project must feel like a coherent product, not a technical demo.

Prioritize:

* strong visual hierarchy
* professional animation timing
* intentional easing
* polished typography
* clear loading states
* clear error states
* understandable AI controls
* a simple end-to-end experience

### Potential Impact

The product should clearly solve a real problem:

> Creating polished product motion design normally requires visual direction, animation experience, and substantial timeline work.

The demo should show that Sequences reduces this work while preserving professional control.

### Quality of the Idea

Differentiate Sequences through:

* a real editable timeline
* structured and reversible AI changes
* inspectable diffs
* targeted revision instead of full regeneration
* product screenshot understanding
* visual self-critique
* humans and AI using the same editor command system

Do not rely only on the claim:

> Generate motion graphics from a prompt.

Existing products already make similar claims.

---

## Build priorities

Prioritize in this order:

1. Reliable prompt-and-image generation.
2. Visually impressive output.
3. A genuine editable timeline.
4. Inspectable AI-generated changes.
5. Natural-language revision.
6. A reliable hosted demo.
7. Clear documentation.
8. Demo-video readiness.

Prefer one excellent workflow over many incomplete features.

Do not prioritize:

* billing
* organization management
* a template marketplace
* complex authentication
* broad collaboration features
* unrelated editor functionality
* large numbers of weak generation modes

---

## Quality bar

The generated motion design should:

* have a clear focal point
* use coherent typography
* avoid text overflow
* use intentional timing and easing
* maintain readable text
* avoid visual collisions
* use consistent spacing
* have purposeful transitions
* feel designed rather than randomly animated

The timeline must contain real editable structure.

A generated result should include appropriate:

* scenes
* layers
* assets
* text
* timing
* keyframes
* easing
* transitions
* animation properties

Natural-language revision should modify the existing composition rather than unnecessarily rebuilding everything.

## Demo experience

Judges may evaluate the project without testing it, so the product must communicate clearly in the video and screenshots.

The hosted demo should still be easy to test.

Preferred flow:

```text
Open demo
    ↓
Load sample project
    ↓
Upload or select image
    ↓
Enter prompt
    ↓
Generate composition
    ↓
Play result
    ↓
Inspect timeline and AI diff
    ↓
Request revision
    ↓
Apply revision
```

Judges should not need to:

* provide an API key
* enter payment information
* rebuild the project
* configure local infrastructure
* complete email verification
* create several external accounts

Keep a prepared sample project and cached fallback result available.

The hosted demo must remain functional through the judging period ending August 5, 2026.
