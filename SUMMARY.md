# Sequences project summary

> Planning handoff for the OpenAI Build Week three-minute demo. This document
> summarizes the repository as it exists on July 20, 2026, including the
> current product, architecture, generation and verification pipeline,
> hackathon constraints, known gaps, and the five finished Showcase films.
>
> The running code, persisted receipts, and pinned HyperFrames contract are the
> source of truth. Some older planning language in `HACKATHON_RULES.md` and
> `design.md` describes a future editable/revision product that is not the
> current MVP.

## Executive summary

Sequences is a local, AI-native motion-design product for making polished SaaS
launch videos. A user writes one prompt, optionally attaches up to four product
images, and clicks **Generate**. Sequences starts a fresh, isolated Codex run,
has GPT-5.6 specialists plan and author a deterministic HyperFrames composition,
strictly verifies the result, performs bounded transactional repairs, and
automatically promotes a passing video into a Sequences-owned watch-only Studio.
The user can then trigger a host-owned render and download both the verified MP4
and the exact source bundle.

The product thesis is:

> One prompt becomes a fresh, professionally directed, server-verified SaaS
> launch film—not a template swap and not an unverified block of generated HTML.

The important ownership boundary is “the model owns taste; the host owns
truth.” GPT-5.6 decides the story, art direction, component architecture,
choreography, camera, and localized creative repairs. The host owns isolation,
file policy, schemas, hashes, deterministic QA, repair adoption, Git promotion,
audio custody, rendering, evidence, and downloads.

The current MVP is intentionally narrower than the long-term “AI motion-design
editor” vision. It has a real HyperFrames player and a simple scene timeline for
playback and scrubbing, but the timeline is watch-only. There is no current
public plan-approval flow, Apply/Reject candidate review, arbitrary editing,
natural-language revision, multi-project management, or hosted no-key judge
experience.

## The most important demo facts

- The required submission video must be under three minutes and have spoken
  narration. The repository records a deadline of July 21, 2026 at 5:00 PM
  Pacific / 8:00 PM Toronto.
- The most impressive result should appear early.
- Generation is real, not a hardcoded video. It is also too slow to wait for in
  a three-minute demo: the production target is roughly ten minutes, while the
  latest persisted passing balanced run took about 20 minutes 57 seconds.
- The website can honestly show the prompt, image attachment, Generate action,
  observed stage progress, the resulting playable composition, its scene strip,
  the render action, and MP4/source delivery.
- The five `Showcase/` films are hand-authored, verified creative controls and
  bounded teaching capsules. They are excellent portfolio/demo assets, but they
  must not be presented as outputs generated by the current one-click pipeline.
- A real current pipeline output exists in the accepted workspace: a 20-second
  fictional Orbit launch film generated by the balanced workflow and promoted
  after contracts, strict QA, layout repair, temporal audit, and compositor
  polish.
- The strongest technological story is not simply “AI made a video.” It is the
  combination of specialist GPT-5.6 creative work, semantic contracts,
  deterministic HyperFrames rendering, quarantined Git custody, visual QA,
  evidence-backed bounded repair, and verified delivery.

## Product experience as implemented

The current website is a single local workspace called `release-a`.

1. The user opens a boot-token URL printed by the local server.
2. A loopback-only session is established.
3. The left panel accepts a plain-language prompt of up to 16,000 characters.
4. The user may attach up to four PNG, JPEG, or WebP reference images. Trusted
   intake decodes each image, limits it to 15 MiB, records immutable dimensions
   and a SHA-256 hash, and stores it under the managed project data directory.
5. Clicking **Generate** always creates a new candidate scaffold and new creative
   custody. It does not edit or inherit the previous composition.
6. The UI streams honest observed events such as preparation, authoring,
   verification, file updates, repair work, applying, completion, or an owned
   failure. It supports cancellation.
7. A candidate is never shown as the accepted result until all required
   contracts and host checks pass. Passing work is automatically promoted; a
   failed or cancelled run leaves the previous video untouched.
8. The right side contains the real `@hyperframes/player`, play/pause, timecode,
   a seekable scrubber, and a scene strip derived from the composition. This is
   a watch-only timeline, not an editor.
9. The user explicitly clicks **Render MP4**. Rendering freezes the accepted Git
   commit, rechecks it, renders it through pinned HyperFrames, performs the
   host-owned audio mix when directed, validates the file, and exposes
   **Download MP4** and **Download source**.

If no generated candidate has been accepted, the product can expose the prepared
`fixtures/release-a` sample. The UI and API distinguish a prepared sample from a
generated candidate; a failed run never silently substitutes the sample.

### HyperFrames versus Sequences

| HyperFrames owns                                               | Sequences owns                                                         |
| -------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Timed HTML composition and exact-time seeking                  | Creative direction and semantic story intent                           |
| Player, editing SDK, clips, tracks, variables, subcompositions | The current watch-only product Studio and future product workflow      |
| GSAP and other supported animation runtimes                    | Commercial taste, component continuity, camera intent, source fidelity |
| Runtime, layout, contrast, motion, and frame diagnostics       | Aggregated QA policy, event-level review, cumulative polish judgment   |
| Rendering mechanism                                            | Isolation, verification custody, promotion, receipts, and delivery     |

Sequences deliberately does not implement a second video renderer or a second
animation DSL. HyperFrames HTML remains the only render source. The semantic
JSON files describe meaning, constraints, and proof; they do not reproduce CSS,
DOM, or tween implementation.

## Generation pipeline and failure ownership

Every run creates a diagnosis packet under
`data/runs/release-a/<run_id>/`. `receipt.json` records the final state, failure
code, failure owner, changed files, QA, repairs, model turns, token/cache usage,
and evidence locations.

| Stage              | What happens                                                                                                                                                                         | Owner and evidence                                               |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------- |
| Queue and prepare  | Checkpoint the request, verify the base commit, create a fresh Git worktree, copy the generic SaaS shell, copy reference images, install and hash-check the bounded skill profile    | Server/Git; `manifest.json`, `prompt.json`                       |
| Preproduction      | In balanced mode, Sol locks `frame.md`, the design capsule, causal `sequence.json`, and the component plan in one bounded turn                                                       | Creative director; turn JSONL/logs and locked artifact hashes    |
| Composition        | Terra alone authors renderable HTML/CSS/JS/assets and `index.motion.json`, implementing the locked story/component handoff                                                           | Compositor; `codex.jsonl`, turn logs, changed files              |
| Post-author policy | Derive the artifact inventory from the Git diff, enforce allowlisted paths, sizes, symlinks and local-assets-only rules, reverify skill hashes, remove the copied skills             | Host policy; receipt `changedFiles`                              |
| Contract gate      | Parse and cross-check story timing, entities, transitions, cameras, design tokens, component states/parts, image bindings, audio cues, and executable motion assertions              | Host/Zod; authored JSON plus contract failure packet             |
| Normalization      | Apply only deterministic fixes with one objectively correct answer, such as missing clip markers, safe GSAP scoping/defaults, font fallback, root paths, and selector reconciliation | Host; candidate diff                                             |
| Verification       | Run pinned HyperFrames lint and strict check with runtime, transition, frame, motion, layout, occlusion, overflow, and WCAG contrast evidence                                        | HyperFrames; `qa/attempt-N/` and snapshots                       |
| Bounded repair     | Run category-owned deterministic fixers, cluster related layout findings, give annotated rendered evidence to the exact compositor thread, and optionally perform one audit polish   | Host + same compositor; repair ledgers, crops, proof comparisons |
| Apply              | Commit the candidate, produce forward and inverse patches, ensure the accepted base is still current, then fast-forward the verified commit                                          | Server/Git; patches and accepted commit                          |
| Render on request  | Freeze the accepted commit, recheck contracts and strict QA, render, mix directed audio, ffprobe metadata, decode first/last frames, and archive source                              | Render host; `artifacts/renders/release-a/`                      |

Repairs are transactional. A repair is adopted only if it strictly improves its
owned category, does not regress another category, and does not alter declared
unchanged proof frames. Otherwise the candidate is restored exactly. Layout
repair is capped at three turns. There is no unbounded rescue ladder.

## GPT-5.6 and Codex workflow

### Production-balanced route

The production default is `SEQUENCES_AGENT_WORKFLOW=balanced`:

- **Sol / medium — preproduction:** owns concept, story hierarchy, frame/design
  direction, semantic beats, component architecture, camera intent, and sound
  direction.
- **Terra / medium — compositor:** is the sole writer of the renderable source,
  product UI, placement, camera implementation, transitions, and motion.
- **Sol / medium — visual auditor:** receives rendered temporal evidence after
  deterministic QA passes. It is read-only and emits a typed audit.
- **Terra / medium — optional audit polish:** the host may resume the exact
  compositor thread for one localized finding, then reruns every lock, contract,
  and strict QA gate.

The stages are sequential because each consumes a hash-locked handoff and the
current cancellation system owns one Codex subprocess at a time. This is not a
voting committee. Later repair also resumes the exact compositor thread and
route; cross-run creative memory is not used.

### Legacy comparison route

`SEQUENCES_AGENT_WORKFLOW=legacy` remains available for controlled comparison.
It uses one fresh all-in-one director, defaulting to GPT-5.6 Luna at high
reasoning. The model and reasoning effort are configurable per server launch,
and every actual turn is recorded in the receipt.

### Bounded author context

The author does not receive the whole repository or an enormous undifferentiated
prompt. The host provides a six-skill, hash-pinned profile:

- `hyperframes`
- `hyperframes-core`
- `hyperframes-creative`
- `hyperframes-animation`
- `hyperframes-keyframes`
- `sequences-saas-launch`

The author reads only selected references through a bounded context gateway.
Prompt plus locked `sequence.json` deterministically selects at most two of the
five Showcase capsules. The receipt records which capsules entered context.
The capsules teach transferable craft; they are never copied as source films or
used as candidate scaffolds.

The current profile assumes local HTML/CSS/SVG, bundled fonts, and GSAP. It does
not currently supply Three.js, Lottie, shaders, premium GSAP plugins, a network,
registry installation, or workflow acquisition during an author run.

## Authored result contracts

A fresh build must replace the generic shell and produce all required design,
story, component, rendering, and motion evidence.

### `frame.md`

The human-readable visual constitution for one film: art direction, hierarchy,
palette/type/geometry intent, motion language, and what the composition should
avoid. The Sequences website design system does not automatically leak into a
generated film; each film owns its own frame.

### `story/design-capsule.json`

A machine-readable design lock containing:

- a catalog, bespoke, or reference-derived origin;
- light/dark basis;
- seven-color palette with contrast expectations;
- bundled Montserrat and IBM Plex Mono roles/weights;
- geometry, density, and composition dialect;
- two to four allowed motion verbs;
- do/avoid rules;
- stable root identity and CSS token bindings.

The four current design foundations are Signal Light, Precision Dark,
Editorial Warm, and Impact Poster. They are starting points, not a host-selected
taste normalizer.

### `story/component-plan.json` v2

The semantic UI architecture. It declares synthetic or reference-derived mode,
tokens, source-image story bindings, and typed components such as app windows,
navigation, buttons, lists, charts, chat, workflows, terminals, and custom
surfaces. Components have stable root IDs, named states, named parts, editable
slots, cause/result interactions, beat ownership, implementation files, and
optional morph anchors. At least one persistent component must carry identity
through the causal product story.

Uploaded screenshots are specifications, not moving screenshot planes. A
reference-derived component plan must bind an image to a native DOM recreation;
rendering the uploaded file directly as the animated product UI is rejected.

### `sequence.json`

The causal story and proof contract. It contains:

- format, frame rate, and target duration;
- concept summary, hierarchy, motion grammar, and rejected choices;
- ordered beats with purpose, claims, sources, semantic entities/parts, proof
  times, implementation files, and optional one-owner camera intent;
- exactly owned transitions between adjacent beats, including entity identity
  for match cuts and morphs;
- narrow semantic declarations for intentional overlaps;
- an optional catalog soundtrack and semantic SFX cue plan;
- optional bounded revision scope for future work.

If the prompt does not specify duration, the author targets 24 seconds. The
normal SaaS-film range is 20–30 seconds.

### `index.motion.json`

Executable evidence assertions such as when an entity appears, relative order,
whether it stays in frame, and whether meaningful motion is present. Its
presence is also disk-level proof that a fresh compositor actually materialized
the build; the starter shell intentionally omits it.

## Motion, component, and sound foundations

The generic `fixtures/saas-shell/` gives every run a contract-valid but neutral
starting point: a host composition, a product-world subcomposition, headline,
window chrome, navigation, stats, chart/activity regions, CSS brand tokens, and
one pointer owner. Shipping it unchanged is a hard failure.

Two candidate-local interaction primitives are implemented:

- A seek-safe typewriter that supports proportional fonts and keeps the caret at
  the measured text edge.
- A normal system-pointer action with measured hotspot geometry and explicit
  approach, settle, press, ripple, release, immediate consequence, and audio
  markers.

The sound contract lets the director choose one hash-pinned soundtrack and up
to 20 semantic cues: typing windows, mouse clicks, pops, wooshes, or
notifications. The host owns the catalog bytes, hashes, gain, fades, looping,
limiting, cue placement, FFmpeg graph, and verification. The current vendored
catalog contains 12 analyzed music beds and five SFX types. A deterministic
beat/energy map is available to the author. `audio: null` means an intentional
silent film.

Still planned or incomplete are generalized camera compilers, path tools, morph
compatibility, transition-velocity checks, narration custody, cue-safe windows,
and full phrase-level audio binding.

## Verification and repair architecture

### Deterministic gate

The pinned HyperFrames gate combines:

- static lint for timing, tracks, timeline registration, selector and tween
  problems;
- strict runtime checks under arbitrary seeking;
- local-asset and browser error checks;
- transition and proof-time snapshots;
- frame-bound and liveness assertions;
- held overlap, occlusion, clipping, and overflow findings;
- WCAG contrast samples;
- motion-sidecar assertions.

A green gate is necessary but not a complete taste verdict. It cannot by itself
judge a semantically wrong camera move, awkward morph, two-owner cursor,
purposeless decoration, weak landing, or several small defects accumulating.
That is why the balanced workflow adds a typed rendered-pixel temporal audit.

### Layout evidence

Repeated layout findings are clustered by root cause instead of being repaired
one screenshot at a time. The host can render a full frame and crop, inspect
the implicated geometry, bind the cluster to semantic beats/entities, and send
the exact evidence back to the compositor. Broad `data-layout-allow-*` escape
hatches are rejected unless they match a narrow declared semantic overlap and
renderer-backed readability evidence.

### Deterministic category fixers

Current class-level fixers include:

- measured WCAG/OKLab contrast repair with cross-pass background memory;
- lint-endorsed `overwrite: "auto"` repair for same-target GSAP tween overlap;
- safe fresh-build normalization for clip markers, selectors, timelines,
  transforms, font fallback, and asset paths.

Creative weakness is not silently normalized. It returns to the exact
compositor with evidence and a narrow allowed-file scope.

### Temporal visual audit

After strict QA, the host builds a typed evidence packet from opening/final
states, beat proof and hold times, transition pre/mid/landed phases, camera
arrival/settle/hold intent, and audio events. Typing reserves start/mid/end
frames; clicks reserve approach/contact/consequence frames. Transit frames are
labeled so motion is not mistaken for a bad landing. The auditor can pass or
request one localized repair, but cannot suppress deterministic findings or
mutate source.

## Git, persistence, and security custody

- Every Generate begins from the currently accepted commit in a disposable
  candidate worktree.
- The host derives completion and artifact inventory from files and Git—not
  from the model’s final prose.
- Only an allowlisted candidate diff can be accepted. Skills are protected,
  hash-reverified, and removed before commit.
- Promotion records candidate and accepted commits plus forward and inverse
  patches. A stale base cannot be silently merged.
- Runs and renders survive as durable JSON receipts and evidence packets.
- The local server binds only to `http://127.0.0.1`, requires the exact Host and
  Origin, uses a boot token, HttpOnly SameSite session cookie, CSRF token, signed
  static file URLs, `no-store`, `nosniff`, and no-referrer headers.
- On Windows the Codex subprocess uses `danger-full-access` because supported
  `workspace-write` installations can silently become read-only. The safety
  boundary is the disposable worktree plus the host-accepted allowlisted Git
  diff. Other platforms default to `workspace-write`.
- Codex API keys and OpenAI API keys are explicitly forbidden in the author
  environment. The product expects `codex login` using ChatGPT.

## Render and delivery

Rendering is intentionally human-triggered and serial per project.

1. Freeze the accepted Git commit into an immutable render worktree.
2. Re-read and validate `sequence.json`, audio direction, and motion evidence.
3. Rerun complete strict HyperFrames QA against the frozen source.
4. Archive the exact accepted commit as a ZIP source bundle.
5. Render with HyperFrames using one worker and the selected quality.
6. If sound is directed, mix the verified catalog bed and semantic SFX with
   FFmpeg, replacing the silent master atomically.
7. Use ffprobe to verify codec, dimensions, frame rate, duration, size, and
   expected audio presence.
8. Decode first and last frames to prove the MP4 is playable at both boundaries.
9. Publish download URLs only after verification completes.

Cancelled renders expose no partial output. Failed renders preserve evidence.

## Showcase collection overview

All five finished Showcase packages are 1920×1080 at 30 fps and include source,
renders, contact sheets, audio where applicable, manifests, and verification
evidence. Four use the current HyperFrames project shape; the historic Slack
golden predates the current `data-*` contract and uses a deterministic paused
GSAP timeline with a local Puppeteer renderer.

The Showcase corpus serves three purposes:

1. Finished portfolio material for the hackathon story.
2. Creative controls that define the motion-quality bar.
3. Compact, hash-pinned teaching capsules selected for new author runs.

### Suggested two-on-website / three-mid-demo split

This is a natural planning split, not a product requirement:

- **Website:** Sequences Abstract Ad and Sequences Recommendation Ad. Both sell
  Sequences directly and show two very different brand/story approaches.
- **Mid-demo:** Slack Ad, ChatGPT Ad, and ChatGPT Native Story. Together they
  quickly establish the inherited craft benchmark, bold launch grammar, and
  authentic code-native product interaction work that informed Sequences.

If the demo needs the most immediately understandable product-UI film on the
website, ChatGPT Native Story can replace Recommendation. The critical honesty
point remains: these are hand-authored Showcase controls, not live generated
examples.

## Showcase 1 — Slack Ad: “From noise to momentum”

**Package:** `Showcase/slack-ad/`
**Duration:** 28.0 seconds
**Engine:** Historic deterministic GSAP/Puppeteer film
**Role:** Original golden craft benchmark and predecessor proof

### What happens

1. Six disconnected tools, files, calls, tasks, and notifications pile into the
   frame under **“Too many tools.”** and **“Too many handoffs.”**
2. The crowded system stalls on **“No momentum.”**
3. Every card and ping collapses into one point; a shockwave fires and the
   official Slack mark assembles from the mess.
4. **“Work shouldn’t feel this messy.”** types under the mark, then a pointer
   clicks it.
5. On a shared desktop, a **Create a channel** modal appears. The user types
   `launch` and clicks Next.
